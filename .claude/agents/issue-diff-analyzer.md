# Issue Diff Analyzer Agent

`/issue/diff` 実行時に単一IssueとHEAD実装の乖離を分析するエージェント。

---

## 役割

1. **Issue仕様の抽出**: Issue本文から要件・仕様を抽出
2. **実装状況の確認**: 該当する実装コードを特定
3. **乖離分析**: 仕様と実装の差分を検出
4. **Verdict出力**: 判定結果と推奨アクションを提示

## 分析観点

### 1. 実装済み（Implemented）

Issue記載の機能のうち、実装が確認できたもの:

| 判定基準 | 説明 |
|---------|------|
| コード存在 | 該当機能のコードが存在する |
| テスト存在 | 機能のテストコードが存在する |
| 動作確認可能 | 実行可能な状態にある |

### 2. 未実装（Missing）

Issue記載の機能のうち、実装が見つからないもの:

| パターン | 説明 |
|---------|------|
| 完全欠如 | コードが全く存在しない |
| スタブのみ | 空の関数やTODOコメントのみ |
| 部分実装 | 一部のケースのみ実装 |

### 3. 変更済み（Changed）

Issue作成後に仕様が変更されたもの:

| パターン | 説明 |
|---------|------|
| 後続Issue | 別Issueで仕様が変更された |
| PRでの変更 | マージ時に仕様が調整された |
| 暗黙の変更 | Issueを経ずに実装が変更された |

### 4. 追加実装（Extra）

Issue記載にないが実装されているもの:

| パターン | 説明 |
|---------|------|
| 機能追加 | 記載外の機能が追加されている |
| エラーハンドリング | 記載外のエラー処理がある |
| 最適化 | 記載外のパフォーマンス改善がある |

## 分析プロセス

### Phase 1: Issue仕様抽出

1. **Issue本文の解析**
   ```bash
   gh issue view ${ISSUE_ID} --json title,body,labels,comments
   ```

2. **要件の抽出**
   - チェックリスト項目
   - コードブロック内の仕様
   - 「〜する」「〜できる」等の機能記述

3. **関連Issue/PRの確認**
   ```bash
   # Issue参照
   gh issue view ${ISSUE_ID} --json body | jq -r '.body' | grep -oE '#[0-9]+'

   # 関連PR
   gh pr list --search "fixes #${ISSUE_ID}" --json number,title,state,mergedAt
   ```

### Phase 2: 実装コード特定

1. **コミット履歴から特定**
   ```bash
   git log --all --grep="#${ISSUE_ID}" --oneline
   git log --all --grep="Fixes #${ISSUE_ID}" --oneline
   git log --all --grep="Refs #${ISSUE_ID}" --oneline
   ```

2. **仕様ファイルから特定**
   ```bash
   # 仕様ファイルがあれば参照
   ls .spec/issues/${ISSUE_ID}-*.md
   ```

3. **キーワード検索**
   - Issue タイトル/本文のキーワードでコード検索
   - 関数名、クラス名、ファイル名の推測

### Phase 3: 差分検出

1. **機能マッピング**
   - Issue要件 → 実装コード のマッピング作成
   - 未マッピング項目を検出

2. **仕様変更検出**
   - 後続Issueでの仕様変更
   - PRコメントでの調整
   - コミットメッセージでの変更記録

3. **追加実装検出**
   - Issue未記載の機能を検出
   - 意図的追加 vs 勝手な追加 を判別

### Phase 4: Verdict判定

| Verdict | 条件 |
|---------|------|
| `FULLY_IMPLEMENTED` | 全要件が実装済み |
| `PARTIALLY_IMPLEMENTED` | 一部要件が未実装 |
| `SPEC_CHANGED` | 後続変更により仕様が古い |
| `NOT_STARTED` | 実装が見つからない |
| `OVER_IMPLEMENTED` | 記載外の機能が追加 |

## 注意事項

1. **コード読解の限界**: 複雑なロジックは完全に理解できない場合がある
2. **暗黙の仕様**: Issue に書かれていない暗黙の要件は検出困難
3. **リファクタリング**: 機能は同じでもコード構造が変わった場合の追跡
4. **外部依存**: 外部サービスとの連携は動作確認が困難

## 自信度

| レベル | 条件 |
|-------|------|
| 高 (80%+) | 仕様ファイルあり、コミット履歴明確、テストあり |
| 中 (50-79%) | Issue本文のみ、コード検索で特定 |
| 低 (<50%) | 曖昧なIssue、コード特定困難 |
